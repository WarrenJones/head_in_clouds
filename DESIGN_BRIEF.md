# Design Brief: Head in the Clouds / 云上心事 v2.4

> 用途：给设计师、Figma Make、图像生成工具同步 Stage 1 设计方向。  
> 对应 PRD：[PRD.md](PRD.md) v2.4 · 2026-05-20。  
> 强约束：不要自由发挥产品结构；必须遵守 PRD 的状态机、权限矩阵、离线优先和分享落地页逻辑。

---

## 1. 一句话定义

Head in the Clouds / 云上心事是一款飞行场景里的情绪 micro app。用户可以先写下一句话，立即生成私人 Cloud Card；航班信息可以稍后补，只有发布到同班机、评论和接收同班机新笔记通知时才需要验证航班。

这版设计的核心不是“人在飞机上会主动想起 App”，而是：

```text
飞行前锚定，表达先行，验证后置，飞行后靠卡片扩散和同班机召回。
```

---

## 2. 设计目标

### 2.0 表达先行，验证后置

航班验证是同班机社交的门槛，不是情绪表达的门槛。

必须让用户理解：
- 可以不扫描登机牌，先写一句。
- 可以不验证航班，先生成并分享私人 Cloud Card。
- 只有发布到同班机、评论同班机笔记、接收同班机新笔记通知时，才需要验证航班。

默认主路径：

```text
写下这一趟 → 生成私人明信片 → 保存/分享 → 可选验证航班并发布到同班机
```

设计上不要让“扫描登机牌”看起来是第一步阻塞。

### 2.1 Cloud Card 是主动获客武器

卡片必须像一张值得被收藏和转发的诗意视觉，不像 App 截图。

默认方向锁定为 **复古明信片 / 票根信纸**：深蓝底 + 米黄纸张 + 航班号信息 + 大字句子 + 坐标路线。这个方向比照片叠字更稳，因为它不依赖用户上传照片或写出高诗意文案。

必须包含：
- 飞行路线 + 起降城市的极简线条 / 坐标感。
- 摘取笔记里最精彩的一句，作为大字排版主角。
- 航班号、日期、主题词作为小号 metadata。
- 左下角极小 app icon / watermark。
- 极低调可识别入口，指向“同班机笔记”页面。

禁止：
- “立即下载”CTA。
- App Store badge。
- 手机 UI 截图式分享图。
- 把登机牌原图做成视觉主角。

模板优先级：
- 默认：`boarding_postcard`，承接所有内容质量，尤其是普通 UGC。
- 次选：`route_poem`，适合极短、克制、偏诗意文本。
- 条件推荐：`cloud_window` / 照片叠字，只有当用户文案和照片情绪一致时才推荐。

### 2.2 写作门槛降到一句话

最小可发布单元是一句话。短内容不是未完成状态。

06 Compose 必须优先呈现：
- 极简版：`这次飞行，我只想说：___`
- 模板版：`飞往 ___，因为 ___，起飞时我 ___`
- 语音版：按住说话 → 转文字 → 用户确认即发

不要用 200 字门槛、长文本空白页或“内容太短”提示吓退用户。

### 2.3 登机通知是激活触发

用户可以手动录入航班号和日期，Stage 1 不依赖 flight tracking API。

通知文案：

```text
你要登上 MU5301 了，今天有什么想留下来的？
```

点击通知后直接进入 06 Compose，并预填航班上下文。

### 2.4 离线优先是生死线

国内航班 WiFi 覆盖弱，首次体验不能出现“写了但发不出去、内容丢了”。

设计必须展示：
- `草稿已保存`
- `已离线保存，落地后自动同步`
- `同步失败，内容已保存 · 重试`

### 2.5 同班机新笔记通知是留存钩子

用户发布后 24-48 小时内，如果同班机有人新发笔记，推送：

```text
MU5301 上昨天有人留下了一条笔记
```

打开后进入 09 Flight Space，并定位到新笔记。

### 2.6 视觉系统不能全深色

深色 UI 有品味，但在中国主流 App 语境里会显得小众、海外、设计师向。设计策略是：

- 深色只做氛围层：启动页背景、卡片外框、少量沉浸页。
- 核心操作区用米白 / 纸张 / 浅色卡片承载：写作编辑器、设置、表单、同班机笔记。
- 阅读区要接近微信 / 小红书用户熟悉的轻背景，不做全局黑底。
- 微信绿 `#07C160` 只能出现在微信登录按钮，不能用于账号状态卡、成功状态大背景或主品牌色。

### 2.7 真实 UGC 质量兜底

不要假设真实用户都会写出“上次坐这趟还是去年”这种高质量文案。产品必须承接普通文本：

```text
飞机晚点了，累死。
终于落地。
这趟航班太吵了。
```

这类文本不适合金色云海、舷窗日落、宏大诗意背景。默认 `boarding_postcard` 应该把它们处理成 flight note / deadpan / 票根备注，而不是强行煽情。

必须提供一个低情绪密度模板方向：
- 名称：`flight_log` / `飞行日志`
- 视觉：深色票根或简洁便签，不要诗意大图。
- 适用：延误、疲惫、吐槽、普通记录。

### 2.8 公开隐私边界

全 app 公开区域禁止显示具体座位号。不要出现 `14A` / `21C` / `7F` / `33B`。

替代身份：
- `靠窗的人`
- `通道旁的人`
- `同机乘客`
- `前舱附近的人`

这个约束适用于 09 Flight Space、10 Discovery、11 Detail、16 Shared Card Landing，以及评论列表。

## 3. 核心内容模型

### 3.1 CloudPost

每条 Post 必须包含：
- 1 张系统生成的 Cloud Card。
- 1 句话到 200 字正文。
- 1 个输入模式：`one_line` / `template` / `voice_transcript` / `free_text`。
- 1 个 headline quote，用于卡片大字。
- 1 个心境标签。
- 航班元数据：可选。未验证时显示“航班待确认”或用户手动输入的路线；验证后显示航班号、日期、出发地、目的地。
- 发布范围：`private_card` / `same_flight`。
- 离线同步状态：`local_only` / `syncing` / `synced` / `sync_failed`。

可选：
- 0-3 张随行照片，只能是窗外、机场、座位、目的地等氛围照片。

不允许：
- 登机牌原图公开展示。
- 视频。
- 外链。
- 多图无限相册。

### 3.2 评论规则

- 本航班：可看评论，可写评论。
- 同旅程 / 同目的地 / 热点：只读，不可评论。
- 11B Discovery Detail 不得出现评论按钮、评论数或评论输入框。

## 5. 关键屏设计要求

### 5.1 06 Compose

页面标题：

```text
这次飞行，我只想说：
```

默认输入框 placeholder：

```text
比如：我把没有说出口的话，带过了云层。
```

顶部航班状态 chip：

```text
未添加航班 · 稍后补
MU5301 · SHA → CTU
MU5301 · 已验证同班机
```

底部操作：
- 主 CTA：`生成私人明信片`
- 次入口：`添加航班，解锁同班机`

模板卡片：
- `飞往 ___，因为 ___，起飞时我 ___`
- `如果能带一样东西上飞机，是 ___`
- `落地以后，我希望 ___`

语音入口：
- 按住说话。
- 转写后显示用户可编辑文本。
- 成功后音频不作为内容资产展示。

### 5.2 07 Cloud Card Studio

Cloud Card 的视觉等级必须高于普通 App UI。它是可传播资产，不是屏幕截图。

三个模板方向：
- `boarding_postcard`（默认）：深蓝底 + 米黄票根信纸 + 邮戳 + 坐标航线 + 大字 quote。适配普通 UGC。
- `route_poem`：极简路线、城市坐标、留白、大字 quote。适配诗意短句。
- `cloud_window`：舷窗光线、云层颗粒、电影感。只在用户文案和照片匹配时推荐。
- `flight_log`：深色票根 / 简洁便签 / 低情绪密度记录。适配延误、疲惫、吐槽和普通飞行日志。

卡片样例文案：

```text
我把没有说出口的话，带过了云层。
```

小字 metadata：

```text
MU5301 · SHA → CTU · 2026.05.19 · 奔赴
```

低质量文案兜底样例：

```text
飞机晚点了，累死。
```

这类内容在卡片上应像一条克制的飞行备注，不要强套日落云海。

未验证状态：
- 仍然允许生成卡片。
- metadata 显示 `航班待确认` 或用户手填路线。
- 发布区主 CTA 为 `保存 / 分享私人卡`。
- 次 CTA 为 `验证航班并发布到同班机`。

### 5.2.1 08 Publish / Share

08 需要拆成两个状态：

`Private Share`
- 不要求航班验证。
- CTA：`保存 / 分享私人卡`。
- 提示：`私人卡不会进入同班机。`

`Same Flight Publish`
- 必须先完成 02-05 航班验证。
- CTA：`发布到同班机`。
- 提示：`验证航班后，同班机的人才能看到和留言。`

### 5.3 15 Flight Reminder Setup

必须低摩擦：
- 航班号输入框默认大写。
- 日期默认今天。
- 起降城市可选，不强制。
- 不展示复杂航班状态。
- 主 CTA：`设置提醒`。
- 次 CTA：`现在就写一句`。

### 5.4 16 Shared Card Landing

这是分享图的落地页，不是下载页。

必须有：
- 原 Cloud Card 大图。
- 同班机笔记 1-3 条预览；没有同班机则展示同航线 / 同目的地精选。
- CTA：`看同班机笔记`。
- CTA：`我也飞过 / 下次也要飞这条航线` → 15。

禁止：
- 首屏 App Store 下载按钮。
- 强营销文案。
- 大面积产品功能介绍。

## 6. 视觉方向

### 情绪关键词

云层、舷窗、起飞前的光、凌晨机场、机舱暗蓝、胶片、票根、邮戳、私密日记、短暂同路人、坐标、航线、静默广播。

### 颜色

- 氛围背景：黎明前深蓝灰，不要纯黑。
- 核心操作 / 阅读面：纸张米白、浅灰蓝、低饱和暖白。
- 暖色：舷窗金、云白、票根米黄。
- 点缀：低饱和蓝、灰绿、褪色橙，不要紫色 AI SaaS 风。
- 微信绿 `#07C160`：只能用于 14 的微信登录按钮。

### 字体感觉

- 英文标题像电影片名。
- 中文标题要克制、暧昧、有私密感。
- 正文像诗句/日记，不像 SaaS 表单。
- 航班号、坐标、日期用等宽字体。

### 组件原则

- Cloud Card 要像实体卡、唱片内页、诗歌海报。
- 票证识别要像“解锁”，不是“上传文件”。
- 评论区像同一趟飞行里的人留短句，不像微博评论区。
- 通知不是运营 push，而是“这一趟飞行”的提醒。

## 8. 设计验收口径

设计不是“屏幕画完”就通过。必须回答：

- Cloud Card 单独拿出来，是否值得用户发朋友圈？
- 06 Compose 是否让用户 60 秒内完成一句话？
- 离线状态是否让用户明确知道内容不会丢？
- 15 的提醒设置是否足够轻，不像航班管理工具？
- 16 是否先进入同班机语境，而不是粗暴下载页？
- 11A / 11B 的评论权限边界是否一眼可见？
- 普通 UGC 如“飞机晚点了，累死”是否也能被卡片承接，而不是审美坍塌？
- 首屏 00a 是否让用户立刻理解“原来飞行可以这样被记录”？
- 用户是否能不扫描登机牌，直接写一句并生成私人卡？
- 公开区域是否完全没有具体座位号？
