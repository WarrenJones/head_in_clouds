> Stage 1 primary deliverable is iOS App. Web/PWA is only auxiliary for shared landing, marketing, or testing fallback.
>
> China market production rule: do not put the primary user-data path on Supabase Cloud or client-direct PostHog Cloud. Use a China-region backend and a first-party event ingestion API. PostHog may remain an internal downstream sink only after server-side relay.
>
> Do not put secrets in this file. Store secrets in Xcode build settings, CI secrets, backend secret manager, provider dashboards, or `.env.local` files excluded by git.

## 4. Production Data Model Draft

The schema is provider-agnostic. Implement it on the selected China-region database, not on Supabase Cloud for production.

| Table | Purpose | Key fields | Access rule |
|---|---|---|---|
| `accounts` | Guest and upgraded accounts | `id`, `device_id_hash`, `wechat_union_id_hash`, `wechat_open_id_hash`, `phone_hash`, `created_at`, `deleted_at` | User can read/update own account only; provider hash is unique for dedup/merge; delete is soft-delete with 30-day recovery. |
| `flight_contexts` | Unverified flight metadata | `id`, `user_id`, `flight_number_hash`, `departure_date`, `route`, `verification_status` | Owner-only until verified/published. |
| `flight_proofs` | Verified proof record | `id`, `user_id`, `flight_context_id`, `method`, `source_image_hash`, `redacted_image_path` | Owner-only; trusted server creates/updates. |
| `cloud_posts` | Private cards and same-flight posts | `id`, `user_id`, `flight_context_id`, `flight_proof_id`, `publish_scope`, `text`, `headline_quote`, `template_id`, `offline_status` | Private scope owner-only; same-flight public read through safe view/API. |
| `comments` | Same-flight comments | `id`, `post_id`, `user_id`, `text`, `created_at` | Insert only if user has verified proof for same flight/date. |
| `reports` | UGC reports | `id`, `reporter_user_id`, `target_type`, `target_id`, `reason` | Reporter can create; admin/service reviews. |
| `blocks` | User blocks | `id`, `user_id`, `blocked_user_id` | Owner-only. |
| `notification_jobs` | Boarding/same-flight notifications | `id`, `user_id`, `flight_context_id`, `type`, `scheduled_at`, `status` | Owner read; trusted server sends. |
| `subscriptions` | IAP verification records | `id`, `user_id`, `transaction_id`, `product_id`, `plan`, `amount`, `currency`, `environment`, `status` | Trusted server writes after StoreKit receipt/transaction validation; app never writes paid status directly. |
| `event_logs` | First-party analytics event store | `id`, `user_id_hash`, `event_name`, `properties_json`, `created_at`, `client_time`, `app_version` | Server write; internal analytics read. |

## 6. Security Rules

- Do not commit `.env.local`, `.p8`, provisioning profiles, DB credentials, WeChat AppSecret, SMS access keys, analytics server keys, Sentry auth tokens, or object storage secret keys.
- Client-side app may only hold client-safe public config: API base URL, Sentry DSN if accepted, and other explicitly public identifiers.
- Event properties must not contain raw email, full phone number, password, ticket number, ID/passport number, raw OCR text, exact public seat number, or payment card data.
- Public UI must not show exact seat numbers. Use “靠窗的人”, “通道旁的人”, “同机乘客”, or “前舱附近的人”.
- Raw ticket images must be deleted after extraction unless the user explicitly stores a redacted proof for themselves.
- Analytics relay must sanitize events before optional downstream forwarding. App must not send flight/emotion/phone identifiers directly to an overseas analytics endpoint.
